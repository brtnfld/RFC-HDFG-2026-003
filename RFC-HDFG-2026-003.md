# RFC-HDFG-2026-003: In-File Blob Configuration Storage for Filters

**Status:** Draft
**Author:** M. Scot Breitenfeld (brtnfld@hdfgroup.org)
**Depends on:** RFC-HDFG-2026-001 (string-based filter configuration)
**Target release:** TBD
**Date:** 2026-07-13

---

## 1. Motivation

RFC-HDFG-2026-001 introduced string-based filter configuration
(`H5Pappend_filter`, `H5Z_class3_t`, `set_config`/`get_config` callbacks)
and raised the practical configuration-size limit from 256 `cd_values`
elements (~1 KB) to 4096 characters via `H5Z_CONFIG_STRING_MAX`. This is
sufficient for most filters but not for all.

Two use cases from LibPressio (raised by @robertu94 during the community
review of RFC-HDFG-2026-001, HDFGroup/hdf5_plugins#255; see
`LIBPRESSIO-FEEDBACK.md` in the RFC-HDFG-2026-001 repository) remain
unaddressed:

### Use Case A â€” SZ4 JIT compiler (primary; fundamental size limit)

SZ4 features a just-in-time compiler for fast user-defined modules. When
used, it needs to store pre-processed source code with the dataset â€”
potentially multi-megabyte. There is no external dataset to reference; the
binary blob must travel with the dataset and be recovered on open. This is
a pure size limitation that `cd_values` and the 4096-byte string limit
cannot address.

### Use Case B â€” ROIBIN-SZ spatial mask (secondary; reference to another dataset)

ROIBIN-SZ stores a binary mask indicating which values use lossless
compression. The preferred design is for this mask to live as a
user-visible HDF5 dataset, with only a reference (a small, fixed-size byte
sequence) stored in the filter configuration. The filter dereferences the
reference at open time to recover the mask.

Both cases are solved by the same mechanism: a binary blob, opaque to the
library, stored alongside the dataset's pipeline message and handed back
to the filter verbatim at open time. This RFC specifies that mechanism as
a single, complete feature: the callback signatures, the on-disk format,
the public creation API, and the parallel I/O, `H5Pencode`/`H5Pdecode`,
and `h5repack` semantics needed to use it correctly.

## 2. Relationship to RFC-HDFG-2026-001 and `H5Z_class3_t`

RFC-HDFG-2026-001 does **not** reserve any placeholder fields in
`H5Z_class3_t` for this RFC. An earlier draft of RFC-HDFG-2026-001
reserved three `void *` fields (`write_blob`/`read_blob`/`close_blob`) for
later activation by this RFC; that approach was dropped (see
RFC-HDFG-2026-001, commit `db64309`) because pre-reserving an opaque
placeholder does not avoid the struct-version bump this feature needs once
real callback signatures are decided â€” it only defers the decision.

Consequently, this RFC's relationship to `H5Z_class3_t` depends on timing:

- **If RFC-HDFG-2026-001 has not yet shipped** when this RFC is adopted,
  the three blob callbacks (Â§3) are added directly to `H5Z_class3_t`,
  exactly as if they had been part of RFC-HDFG-2026-001 from the start. No
  new struct version is required.
- **If `H5Z_class3_t` has already shipped**, adding these fields requires
  a new `H5Z_class4_t` (or equivalent extension mechanism, e.g. the
  `H5Z_CLASS_T_VERS_MAX` pattern already documented in
  RFC-HDFG-2026-001 Â§4.2). This RFC does not resolve that question in
  advance; whoever finalizes this RFC should check the shipped status of
  `H5Z_class3_t` at that time and adjust Â§3 accordingly.

The callback types are defined in Â§3.1.

## 3. Proposed Design

### 3.1 Blob Callback Typedefs and `H5Z_class3_t` Fields

```c
/* Fixed-size on-disk locator for a blob's global-heap object.
 * Opaque to plugin authors; passed back unchanged between write_blob
 * and read_blob. */
typedef struct H5Z_blob_loc_t {
    haddr_t addr;   /* global heap collection address */
    size_t  idx;    /* object index within the collection */
} H5Z_blob_loc_t;

/* Called at H5Dcreate time to persist a blob alongside the pipeline
 * message.  If NULL, the library uses its default H5HG-based writer.
 * loc_out identifies the stored object for the later read_blob call. */
typedef herr_t (*H5Z_write_blob_func_t)(hid_t            file_id,
                                        const void       *buf,
                                        size_t            size,
                                        H5Z_blob_loc_t   *loc_out);

/* Called at H5Dopen time to recover a blob written by write_blob.
 * If NULL, the library uses its default H5HG-based reader.
 * *buf_out is allocated by the callback (or the library, in the
 * default case); the caller releases it via close_blob. */
typedef herr_t (*H5Z_read_blob_func_t)(hid_t            file_id,
                                       H5Z_blob_loc_t    loc,
                                       void            **buf_out,
                                       size_t           *size_out);

/* Called at H5Dclose time to release the in-memory blob buffer
 * returned by read_blob.  If NULL, the library calls free(). */
typedef herr_t (*H5Z_close_blob_func_t)(void *buf, size_t size);
```

Added to `H5Z_class3_t` (or `H5Z_class4_t`, per Â§2):

```c
H5Z_write_blob_func_t       write_blob;  /* NULL-able */
H5Z_read_blob_func_t        read_blob;   /* NULL-able */
H5Z_close_blob_func_t       close_blob;  /* NULL-able */
```

All three fields are NULL-able. When NULL, the library uses a default
implementation built on `H5HG` (the existing global-heap mechanism already
used for large variable-length attribute data): `H5HG_insert` for writing,
`H5HG_read` for reading, and `free()` for closing. Plugins implement these
callbacks only when they need custom on-disk layout â€” for example,
alignment requirements for GPU DMA transfers directly out of the file
buffer. `H5Z_blob_loc_t` is an opaque locator handed unchanged from
`write_blob` to the pipeline-message encoder and later from the decoder to
`read_blob`; plugin authors do not construct or interpret its fields.

The three callbacks are independent of `set_config`/`get_config`: a
filter's parameter string continues to carry small, human-readable
configuration, while the blob carries the large, opaque payload. A filter
that needs both derives any small values it needs from dataset properties
via the existing `set_local` callback, or encodes them as a short header
inside the blob itself; the blob content is entirely filter-defined bytes,
so this requires no new library mechanism.

Add `has_blob_callbacks` (a `bool`) to `H5Z_class_info_t`
(RFC-HDFG-2026-001 Â§4.3.2) so `H5Zget_filter_class_info` can report
whether a registered filter implements the blob callbacks.

### 3.2 New Internal State: `H5Z_filter_info_t` Additions

```c
/* H5Zprivate.h -- internal only; not part of the public API */
struct H5Z_filter_info_t {
    /* ... existing fields ... */
    void            *aux_data;  /* in-memory blob; NULL if this filter has none */
    size_t           aux_size;  /* byte length of aux_data                     */
    H5Z_blob_loc_t   aux_loc;   /* on-disk locator; undefined until written    */
};
```

### 3.3 On-Disk Format

The current `H5O_PLINE_VERSION_2` per-filter layout:

```
[id:2][flags:2][cd_nelmts:2][cd_values:cd_nelmtsĂ—4][name:variable]
```

This RFC's format adds a `has_aux` bit after the name and, when set, the
fixed-size `H5Z_blob_loc_t` encoding (address plus heap-object index, the
same fields `H5HG_t` already carries internally for other large-object
references):

```
[id:2][flags:2][cd_nelmts:2][cd_values:cd_nelmtsĂ—4][name:variable]
[has_aux:1][aux_loc:variable if has_aux]
```

**Version number.** This RFC needs a pipeline-message format bump.
RFC-HDFG-2026-001 introduces no on-disk format change, so this RFC is free
to claim `H5O_PLINE_VERSION_3` â€” **unless** some other RFC has claimed
that number first by the time this RFC is finalized (RFC-HDFG-2026-001
mentions two other still-hypothetical format extensions â€” name-only
pipeline entries for soft-failing optional name-based filters, and typed
per-slot `cd_values` storage â€” that were at one point informally
associated with "version 3"; neither is committed to a specific version
number). Whoever finalizes this RFC should check for version-number
conflicts with any other pipeline-message format work in flight at that
time and claim the next free version.

The blob bytes themselves are stored as a global-heap (`H5HG`) object; the
pipeline message records only the locator, not the bytes inline.
`H5O_pline_ver_bounds[]` maps `H5F_libver_t` to the pipeline message
version; a DCPL with a blob-bearing filter requires a libver upper bound
that admits this RFC's pipeline-message version, exactly as any other
version-gated format feature does.

Runtime I/O is unaffected: `H5Z_pipeline`, the hot path for chunk I/O,
touches none of this. The blob is configuration data, loaded once at open
time; compression/decompression performance is unchanged.

### 3.4 New Public API: `H5Pappend_filter_blob`

```c
herr_t H5Pappend_filter_blob(
    hid_t         plist_id,
    H5Z_filter_t  id,
    unsigned      flags,
    const void   *buf,
    size_t        size
);
```

**Description.** Appends filter `id` to the pipeline on `plist_id`,
exactly as RFC-HDFG-2026-001's `H5Pappend_filter` does, and attaches
`buf`/`size` as the filter's blob. The bytes are opaque to the library;
they are handed to the filter's `write_blob` callback (or the default
`H5HG` writer) at `H5Dcreate` time and recovered via `read_blob` (or the
default reader) at `H5Dopen` time. `set_config` is not invoked for this
entry point â€” the blob path bypasses the parameter-string layer entirely,
since the whole point of the blob channel is to carry data that does not
fit through it.

**Decision: single entry point, no `H5Z_params_t` variant.** An earlier
draft considered adding `H5Z_PARAMS_BLOB` as a third
`H5Z_params_type_t` tag so that blob configuration could flow through
`H5Pappend_filter`'s tagged union. This RFC does not do so:
`H5Z_params_t` is designed around values that fit comfortably in a
compound literal, and forcing a multi-megabyte buffer through that union
would either double-store the buffer or complicate its lifetime rules for
no benefit. `H5Pappend_filter_blob` is a dedicated function with a
signature suited to its purpose; `H5Z_params_type_t` is unchanged by this
RFC.

**Parameters:**
- `plist_id` â€” A dataset creation property list identifier.
- `id` â€” The filter identifier. The registered filter must implement
  `write_blob`/`read_blob` or accept the library's default `H5HG`-based
  implementation.
- `flags` â€” Same semantics as `H5Pappend_filter`.
- `buf` â€” Pointer to the blob bytes. The library copies these bytes into
  DCPL-owned storage before returning (see Â§4.2); the caller's buffer may
  be freed or reused immediately after the call.
- `size` â€” Length of `buf` in bytes. There is no fixed upper bound
  analogous to `H5Z_CONFIG_STRING_MAX`; the practical limit is available
  memory and, in parallel jobs, the cost of the broadcast described in
  Â§4.1.

**Returns:** Non-negative on success; negative on failure.

**Errors:**
- `H5E_NOFILTER` â€” The filter plugin could not be found or loaded.
- `H5E_BADVALUE` â€” `buf` is `NULL` with nonzero `size`, or `size` is zero
  with non-NULL `buf`.

### 3.5 Dataset Create / Open / Close Lifecycle

**Create** (`H5D__create`), after `H5Z_set_local` runs, for each filter
with `aux_data != NULL`:
1. Call `filter->cls->write_blob(file_id, aux_data, aux_size, &aux_loc)`,
   or the default `H5HG` writer if `write_blob` is `NULL`.
2. Store `aux_loc` in `H5Z_filter_info_t`.
3. Encode the pipeline message at this RFC's format version with
   `has_aux` set and `aux_loc` populated.

**Open** (`H5D__open`):
1. Decode the pipeline message; for each filter with `has_aux`, read
   `aux_loc`.
2. Call `filter->cls->read_blob(file_id, aux_loc, &aux_data, &aux_size)`,
   or the default `H5HG` reader if `read_blob` is `NULL`.
3. Populate `aux_data`/`aux_size` in `H5Z_filter_info_t`.

**Close:** Call `close_blob(aux_data, aux_size)`, or `free()` if
`close_blob` is `NULL`, and clear the fields.

**Delete** (dataset deleted, or the filter removed from a pipeline being
modified): the pipeline message's `delete` handler in `H5Opline.c` calls
`H5HG_remove` on `aux_loc` to reclaim the heap space, analogous to how the
fill-value message's `delete` handler frees vlen data today
(`H5Ofill.c`).

## 4. Resolved Design Decisions

The four open design questions originally raised for this feature are
resolved as follows. (These resolutions were reached during a period when
this design was folded into RFC-HDFG-2026-001; they carry over unchanged
now that the feature is split back out into its own RFC.)

### 4.1 Parallel I/O Write Protocol

**Decision:** Rank 0 writes the blob; its locator is broadcast to all
ranks before the pipeline message is encoded.

`H5HG_insert` has no MPI awareness: it allocates file space and dirties a
metadata-cache entry directly, outside the collective-I/O coordination
that object-header message appends go through. If every rank in a
collective `H5D__create` called `H5HG_insert` independently, each would
create a separate heap object at a different address, with no way to
agree on a single locator for the pipeline message. The protocol:

```c
H5Z_blob_loc_t loc;

if (H5F_HAS_FEATURE(file, H5FD_FEAT_HAS_MPI)) {
    int mpi_rank;
    MPI_Comm_rank(H5F_mpi_get_comm(file), &mpi_rank);
    if (mpi_rank == 0)
        write_blob_or_default(file, aux_data, aux_size, &loc);
    MPI_Bcast(&loc, sizeof(loc), MPI_BYTE, 0, comm);
} else {
    write_blob_or_default(file, aux_data, aux_size, &loc);
}
/* all ranks now share the same locator; proceed to encode the
 * pipeline message collectively */
```

This is the same pattern already used elsewhere in HDF5 for single-writer
allocations under `H5FD_mpio` (e.g., superblock writes). Calling
`H5Pappend_filter_blob` on a handle opened for parallel I/O is legal and
must be followed by a collective `H5D__create`; when the DCPL is built
before the file is opened (the common pattern), the broadcast happens
inside `H5D__create`, invisible to the caller.

The broadcast is cheap regardless of blob size: only the fixed-size
locator (a few dozen bytes) is broadcast, never the blob bytes
themselves, so the cost added to a collective `H5Dcreate` is independent
of blob size.

### 4.2 `H5Pencode`/`H5Pdecode` Policy

**Decision:** `H5Pencode` serializes the blob's bytes inline;
`H5Pdecode` leaves the on-disk locator undefined until the next
`H5Dcreate`.

`H5P__ocrt_pipeline_enc` encodes filter ID, flags, name, and `cd_values` â€”
all in-memory data; no file address appears in an encoded DCPL. Fill-value
encoding follows the same rule: a serialized property must be
self-contained, able to survive a buffer, a process boundary, or a decode
without the original file. `aux_loc` is a derived, file-specific value;
`aux_data` is the canonical in-memory representation.
`H5P__ocrt_pipeline_enc` is extended to append, after each filter's
`cd_values`:

```
[has_aux:1][aux_size:8 if has_aux][aux_data:aux_size bytes if has_aux]
```

`H5P__ocrt_pipeline_dec` reads those fields back into `aux_data`/`aux_size`
and leaves `aux_loc` undefined; a subsequent `H5D__create` using the
decoded DCPL triggers `write_blob` and assigns a real locator in whichever
file it is used with.

Serializing the locator instead of the bytes was rejected: it would make
an encoded DCPL valid only in its originating file, breaking every
`H5Pdecode`-then-`H5Dcreate` workflow in a different file. Returning an
error for blob-bearing DCPLs was also rejected as unnecessarily
restrictive, since `aux_data` is always available in memory after
`H5D__open` or after `H5Pappend_filter_blob`. The tradeoff accepted is
that encoded DCPLs carrying a multi-megabyte blob are correspondingly
large; this is acceptable since `H5Pencode` is not a hot path and callers
invoking it on such a DCPL already know they hold unusual data.

### 4.3 `h5repack` and `H5Pcopy` Behavior

**Decision:** No `h5repack`-specific code is needed; `H5Pcopy` is
extended to deep-copy `aux_data`.

`h5repack` copies a normal (non-external) dataset's DCPL via
`H5Pcopy(dcpl_in)` and does not call `H5HG` functions directly. The
existing lifecycle already makes this work automatically, *provided*
`H5Pcopy` performs a deep copy of `aux_data`:

1. `H5Dopen` on the source populates `aux_data` in the source DCPL's
   filter entry (via `read_blob` or the default reader).
2. `H5Pcopy(dcpl_in)` must `malloc` + copy the `aux_data` buffer for each
   filter that has one, leaving `aux_loc` undefined in the copy (the
   destination address is not yet known).
3. `H5Dcreate` with the copied DCPL in the destination file triggers
   `write_blob` (or the default writer), producing a new heap object at a
   new locator in the destination file. Only the bytes travel; the source
   locator is never copied.

The pipeline message's `copy` callback in `H5Opline.c` is extended
accordingly. When a filter is removed from a pipeline (e.g., `h5repack`'s
`apply_filters()` replacing a filter), the pipeline message's destructor
frees `aux_data` the same way it already must free other per-filter
allocations.

### 4.4 Storage Format: Naked `H5HG`

**Decision:** Blob bytes are stored as a plain `H5HG` global-heap object,
not a new HDF5 object message type.

A new message type (`H5O_MSG_FILTER_BLOB`) would let `h5dump` label the
object (e.g., "4 MB of SZ4 JIT source"), but the payload remains
filter-plugin-specific binary data that no generic tool can render
meaningfully â€” the pipeline message already identifies the filter, which
is the useful part of that label. A new message type also carries an
ongoing maintenance cost (roughly a dozen callbacks: encode, decode, copy,
`raw_size`, reset, free, delete, link, and the pre/copy/post-copy-file
hooks) for marginal tooling value. Existing HDF5 precedent for "large
binary thing referenced from the object header, recovered on demand" is
exactly `H5HG`, already used for large variable-length attribute data. If
a genuine tooling need for a self-describing message type emerges after
deployment, one can be introduced in a later file-format version without
breaking files written under this design â€” the pipeline message's
`aux_loc` field would simply become one of several supported locator
kinds.

**`del` callback obligation:** When a pipeline message is deleted (dataset
deleted, filter removed), `H5HG_remove(file, &loc)` must be called to
reclaim the heap space. This is implemented in the pipeline message
`delete` handler in `H5Opline.c`, analogous to how the fill-value
message's `delete` handler frees vlen data (`H5Ofill.c`).

### 4.5 Use Case B: Reference to Another Dataset

Use Case B (ROIBIN-SZ) stores an HDF5 reference to a user-visible mask
dataset as the filter's blob â€” a small, fixed-size byte sequence rather
than a multi-megabyte payload, but otherwise an ordinary use of
`H5Pappend_filter_blob`. Dereferencing that reference to fetch the mask's
actual contents requires a file handle, which an earlier draft proposed
solving with a new `H5Z_get_file_id()` accessor or an additional
`open_dataset` callback.

**Decision:** Neither is needed. `read_blob` (Â§3.1) already receives
`file_id` â€” required for Use Case A's default `H5HG`-based storage â€” and
that same handle is sufficient for Use Case B. A filter that stores a
reference as its blob dereferences it inside its own `read_blob`
implementation using the supplied `file_id`, fetches the referenced
dataset's contents once at open time, and caches them in filter-private
state (the `aux_data` buffer, or memory it allocates and tracks itself)
for use during subsequent filter-callback invocations. No per-chunk file
access and no additional public API are required; the same callback
signature already specified for Use Case A covers both use cases.

## 5. Under Discussion: Retrieving a Default-Stored Blob

Prototype experience surfaced a gap in the default-storage path: a blob
written through the default `H5HG` writer is currently *write-only*. The
library recovers the bytes into the pipeline at `H5Dopen`, and they
propagate correctly through `H5Pcopy`, `H5Pencode`/`H5Pdecode`, and
`h5repack` â€” but no public API exposes them. An application cannot read a
blob back from a dataset's DCPL, and a filter cannot consume its own blob
unless it implements *both* `write_blob` and `read_blob`: a class that
implements only `read_blob` cannot interpret the default writer's locator,
because `H5HG` is library-private. Both use cases in Â§1 therefore require
custom callbacks today, which makes the default storage path considerably
less useful than it appears.

**Option under discussion:** a public accessor,

```c
herr_t H5Pget_filter_blob(
    hid_t     plist_id,
    unsigned  idx,        /* pipeline index, as in H5Pget_filter2       */
    void     *buf,        /* NULL for a size query                      */
    size_t   *size        /* in: capacity of buf; out: blob byte count */
);
```

following the same size-query convention as
`H5Pget_filter_params_by_idx`. This would let applications and tools
(e.g., an `h5dump -p` annotation that a blob is present and its size)
retrieve default-stored blobs, and â€” notably â€” would let a *filter*
consume its blob without implementing any storage callbacks: the
`set_local` callback already receives the dataset's `dcpl_id` and could
fetch and cache the blob there. That combination (default storage +
`set_local` retrieval) would cover Use Case A with no filter-side storage
code at all.

Alternatives considered but not currently favoured: keeping the status
quo (filters that consume blobs must implement both storage callbacks),
or passing the blob into the filter callback through a new context
parameter (rejected for widening the hot-path signature). No decision is
recorded here; this section exists to solicit review feedback before the
accessor is added to the implementation.

## 6. What Is Explicitly Out of Scope

- Changes to `H5Z_func_t`, the runtime I/O callback used by
  `H5Z_class1_t`/`H5Z_class2_t` filters â€” the hot path is unchanged by

- Changes to `H5Z_class2_t` or `H5Z_class1_t`.
- Blob support for group fractal heap filters.
- Multi-file blob sharing (a blob in file A referenced by a dataset in
  file B).

## 7. Test Plan

These tests exercise `H5Pappend_filter_blob` and the blob on-disk format
(Â§3.3).

- **blob-01** (create/open round-trip): Create a chunked dataset with a
  filter configured via `H5Pappend_filter_blob` using the library's
  default `H5HG`-based storage (`write_blob`/`read_blob` both `NULL`).
  Close and reopen the file; verify the filter's `read_blob` path
  (default reader) returns bytes identical to those originally passed to
  `H5Pappend_filter_blob`.
- **blob-02** (custom callbacks): Repeat blob-01 with a test filter that
  implements non-NULL `write_blob`/`read_blob`/`close_blob`; verify all
  three are invoked at the expected lifecycle points (Â§3.5) and that
  `close_blob` runs exactly once per `H5Dclose`.
- **blob-03** (delete reclaims heap space): Create a blob-configured
  dataset, note the file size, delete the dataset, and verify (e.g., via
  `h5stat` or a subsequent `H5Fget_freespace` check) that the heap space
  is reclaimed rather than leaked.
- **blob-04** (`H5Pencode`/`H5Pdecode` round-trip): Build a DCPL via
  `H5Pappend_filter_blob`, `H5Pencode` it, and `H5Pdecode` the result into
  a new DCPL in a separate process with no access to the original file.
  Verify the decoded DCPL's blob bytes are byte-identical to the
  original, confirming Â§4.2's inline-serialization decision.
- **blob-05** (`h5repack` cross-file copy): Run `h5repack` on a file
  containing a blob-configured dataset; verify the output file's dataset
  reads back correctly and that its blob is stored at a new, independent
  on-disk locator (not the source file's locator), per Â§4.3.
- **blob-06** (oversized default-storage blob): Configure a filter via
  `H5Pappend_filter_blob` with a multi-megabyte buffer using the default
  `H5HG`-based storage; verify create/open succeeds and the runtime chunk
  I/O path is unaffected (compression throughput on the same file is
  unchanged relative to an equivalent dataset without a blob).
- **blob-E01** (malformed call): `H5Pappend_filter_blob` with `buf=NULL`
  and `size>0`, or `buf!=NULL` and `size=0`, returns `H5E_BADVALUE` and
  does not modify the DCPL.
- **blob-E02** (unknown filter): `H5Pappend_filter_blob` with an
  unregistered filter `id` returns `H5E_NOFILTER`.
- **par-blob** (rank-0 write and broadcast; requires a Parallel HDF5
  build and at least two MPI ranks): All ranks call
  `H5Pappend_filter_blob` with identical arguments, followed by a
  collective `H5Dcreate`. Verify that exactly one heap object is created
  (not one per rank) and that every rank's copy of the pipeline message
  carries the same locator, confirming the rank-0-write-and-broadcast
  protocol of Â§4.1.
